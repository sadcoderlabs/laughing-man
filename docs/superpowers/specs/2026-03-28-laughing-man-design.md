# laughing-man: Markdown-to-Newsletter CLI Tool

## Overview

A CLI tool that turns a folder of markdown documents into a newsletter: a static archive website on GitHub Pages and email delivery via Resend. Built with Bun + TypeScript + React Email. Designed for open source from day one.

**Repository:** Standalone repo (not inside of-course-i-still-love-you).

**Distribution:** npm package. `npx laughing-man init`. Binary distribution (`bun build --compile`) planned for later once the React Email + Bun compile bug is resolved.

## CLI Interface

```bash
laughing-man init              # Generate config.yaml
laughing-man preview           # Build full site + email HTML, open in browser
laughing-man preview 2         # Preview just issue 2
laughing-man publish 1         # Deploy site to GitHub Pages + send issue 1 via Resend
laughing-man publish 1 --yes   # Non-interactive mode (for CI/GitHub Actions)
```

## Config

Generated by `laughing-man init`. Lives in the user's newsletter directory as `config.yaml`.

```yaml
name: "The Net is Vast and Infinite"
url: "https://thenetisvastandinfinite.com"

issues_dir: ./drafts
theme: default

resend_api_key: "re_xxxxx"
resend_audience_id: "aud_xxxxx"

email:
  from: "Vinta <vinta@thenetisvastandinfinite.com>"
  reply_to: vinta@thenetisvastandinfinite.com

hosting:
  provider: github-pages
  repo: vinta/mensab

subscribe_api_url: "https://newsletter-subscribe.vinta.workers.dev"
```

**Environment variable override:** Any config value can be overridden by an env var. `RESEND_API_KEY` overrides `resend_api_key`, etc. This allows committing `config.yaml` without secrets and injecting them in CI.

A `config.example.yaml` ships with the package for open source reference.

## Content Format

Each newsletter issue is a markdown file with YAML frontmatter:

```markdown
---
title: "創刊號：網路是無限寬廣的！"
issue: 1
date: 2026-03-28
---

Content here in markdown...
```

Single language per issue. No multi-language support in v1.

## User's Directory Structure

After `laughing-man init`, the user's newsletter directory looks like:

```
my-newsletter/
  .laughing-man/         # Tool state (gitignored)
    state.json           # Tracks which issues have been sent
  config.yaml
  drafts/                # Markdown issue files (configurable via issues_dir)
    issue-1.md
    issue-2.md
  output/                # Generated files (gitignored)
    site/                # Static site for GitHub Pages
      index.html
      issues/
        1/index.html
        2/index.html
      subscribe/index.html
    email/               # Email HTML for preview/debugging
      1.html
      2.html
```

Minimal footprint. Only `config.yaml` is scaffolded by `init`.

## Theme System

### Built-in themes

Themes live inside the `laughing-man` package source under `themes/`.

```
laughing-man (package source)/
  src/
  themes/
    default/
      email.tsx          # React Email template for the newsletter email
      web.tsx            # Template for individual issue page
      index.tsx          # Archive/home page listing all issues
      subscribe.tsx      # Subscribe page with form
      assets/
        styles.css       # Web-only styles
```

v1 ships with one theme: `default`.

### Theme resolution

1. Check `./themes/<theme-name>/` in the user's newsletter directory
2. Fall back to `laughing-man`'s built-in `themes/<theme-name>/`

Per-file override: if the user creates `./themes/default/email.tsx`, only the email template is overridden. Everything else still comes from the built-in default.

### Templates

All templates are React Email components (`.tsx`). They receive the rendered markdown HTML and issue metadata as props:

```tsx
// Simplified example of what a theme template receives
interface IssueProps {
  title: string
  issue: number
  date: string
  content: string  // Rendered HTML from markdown
  config: SiteConfig
}
```

## Build Pipeline

Both `preview` and `publish` run this pipeline:

1. Read `config.yaml` (with env var overrides applied)
2. Scan `issues_dir` for `.md` files, parse frontmatter with gray-matter
3. Sort issues by issue number
4. For each issue:
   - Render markdown to HTML (via marked)
   - Pass through theme's `email.tsx` to produce email HTML
   - Pass through theme's `web.tsx` to produce web page HTML
5. Render `index.tsx` with the list of all issues
6. Render `subscribe.tsx` with the `subscribe_api_url` from config
7. Write everything to `output/site/` and `output/email/`

`preview` opens `output/site/index.html` in the browser.

`publish` additionally runs deploy and send steps.

## Publish Flow

`laughing-man publish <issue-number>` does three things in sequence:

### 1. Build

Same pipeline as `preview`. Generates `output/site/` and `output/email/`.

### 2. Deploy site

Pushes `output/site/` to the `gh-pages` branch of the configured GitHub repo.

### 3. Send email

- Reads `output/email/<issue>.html`
- Fetches all contacts from the Resend audience
- Sends via Resend's batch send API (100 recipients per call)

### Safety checks

- **Duplicate send prevention:** Checks `.laughing-man/state.json` for whether this issue has already been sent. Warns and aborts if so.
- **Confirmation prompt:** Shows "Send issue 1 to 47 subscribers? [y/N]" before sending. Skipped with `--yes` flag for CI.

### Sent state tracking

`.laughing-man/state.json` in the user's directory:

```json
{
  "sent": {
    "1": { "date": "2026-03-28", "recipients": 47 }
  }
}
```

### CI support

Designed to run in GitHub Actions:

```yaml
- run: npx laughing-man publish 1 --yes
  env:
    RESEND_API_KEY: ${{ secrets.RESEND_API_KEY }}
```

Env vars override config.yaml values, so secrets stay out of the repo.

## Subscribe System

### Architecture

The newsletter website is static (GitHub Pages), so form submissions need a small backend. A Cloudflare Worker handles this.

### Worker

Source lives in the `laughing-man` repo under `worker/`. Not generated into the user's project.

Endpoints:
- `POST /subscribe` - validates email, adds to Resend audience, returns success/error
- `POST /unsubscribe` - removes from Resend audience

CORS restricted to the newsletter's configured `url`.

The user deploys this worker to Cloudflare Workers (free tier) using `wrangler`, sets `RESEND_API_KEY` and `RESEND_AUDIENCE_ID` as Cloudflare secrets, and puts the worker URL in `config.yaml` as `subscribe_api_url`.

### Subscribe page

The theme's `subscribe.tsx` renders a page with a form that posts to `subscribe_api_url`. Part of the static site, deployed alongside issue pages.

## Key Dependencies

- **commander** - CLI framework
- **@react-email/components** + **@react-email/render** - Email template rendering
- **gray-matter** - YAML frontmatter parsing
- **marked** - Markdown to HTML
- **resend** - Resend API SDK
- **yaml** - config.yaml parsing
- **zod** - Config and frontmatter validation

## Open Source Design Principles

- Provider logic isolated in `src/providers/`. Adding a new email provider (SES, Mailgun) or hosting provider (Netlify, Vercel) means adding a new file, not modifying existing ones.
- No user data in the package. All state lives in the user's directory.
- Config-driven. No hardcoded assumptions about newsletter name, domain, or provider details.
- `config.example.yaml` documents all options.
- Theme system allows customization without forking.

## Out of Scope for v1

- Multi-language support (i18n, language switching, subscriber language preference)
- Binary distribution (`bun build --compile`)
- `eject-theme` command
- Paid tiers or monetization features
- Analytics or open/click tracking
- Double opt-in confirmation flow
- Webhook processing for delivery events
- Multiple newsletter support
